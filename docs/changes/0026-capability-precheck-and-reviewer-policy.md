# 任务前能力预检与条件审查者

## 背景

用户希望后续任务开始前，不只是直接执行，还要先判断是否有更合适的 skills、tools、plugins、connectors、联网搜索或审查者可以辅助。同时，用户希望审查者按风险分级触发，并且当用户明确指定审查者数量或作用时，优先按用户意向执行。

## 改动

- 更新 `AGENTS.md` 的协作与交付偏好。
- 新增“任务前能力预检”规则。
- 明确插件或新 skill 的安装/启用必须先说明用途、风险和成本，并获得用户确认。
- 更新审查者分级规则，补充“用户明确指定审查者数量或作用时，优先按用户意向执行”的条件。

## 行为

- 纯聊天、状态同步、小问题解释：不触发复杂工具搜索或审查流程。
- 较大任务、新功能、API 接入、数据源接入、报告生成逻辑、架构调整、安全/额度相关任务：先做轻量能力预检。
- 普通代码、配置、脚本、测试改动：默认至少 1 个审查者或等价独立自审。
- API、live、架构、安全、数据流、大版本变更：默认 2 个审查者。
- 用户明确指定审查者数量或作用时，在不违反安全与工具边界的前提下优先执行。

## 安全边界

- 不自动安装插件。
- 不自动启用新 connector。
- 不为了低风险小任务反复搜索工具。
- 不因用户指定较少审查者而静默降低高风险任务的安全标准；如果指定强度低于风险需要，先解释并请求确认。

## 后续

未来可以创建项目专用 reviewer skill，例如：

- `ceramic-product-reviewer`
- `ceramic-security-reviewer`
- `ceramic-report-quality-reviewer`
- `ceramic-architecture-reviewer`

这些 skill 可用于固定审查清单，但不等同于自动启动独立子代理。
